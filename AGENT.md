1. アプリの背景・コンセプト
問題意識
ジュニアエンジニアとして 「何か作りたいけど、題材がつまらない（ToDoアプリ等）」 というモヤモヤ。
「ポートフォリオ用の課題」ではなく、
誰かの本当に欲しいアイデアを元に作って実践したい というニーズ。
一方で、
アイデア提供側のインセンティブが弱くなりがち（わざわざ書くメリットが薄い／革新的アイデアは出しづらい）という課題も意識。
seed のコンセプト
名前：[seed]
「アイデアの種（seed）」→ エンジニアが育ててプロダクトにするイメージ。
目的：
アイデア提供側：アイデアの叩き台・検証の場を得る。
エンジニア側：実践的なポートフォリオづくりの場を得る。
将来的には：
ここで作られたソリューションの「採用回数」「評価」などを可視化して、
就活やキャリアで評価される仕組みにしたい。
アイデアそのものにも収益が発生する世界（閲覧数・インプレッション等に応じた還元）を目指す。
2. ユーザー像とインセンティブ
想定ユーザー
アイデア提供者
エンジニア／非エンジニア問わず
「こういうツールあれば便利」「この課題、誰か解いてほしい」と思っている人。
ビルダー（エンジニア）
ジュニア〜中級想定。
「就活・転職に使える本気のポートフォリオ」を作りたい人。
将来の追加ペルソナ
企業の採用担当・リクルーター
実務に近いアウトプットを見て人材を探したい人。
インセンティブ設計（方針レベル）
エンジニア側：
実在の人のニーズを起点にしたプロダクトを作れる。
「ソリューションの採用回数」「評価」「使用技術」などがプロフィールとして可視化され、実践ポートフォリオになる。
アイデア提供側：
アイデアごとに
どれだけ見られたか（閲覧数）
何人が「作ってみたい」と手を挙げたか
実際に何個ソリューションが生まれたか
が見える → アイデアの検証・実績化ができる。
将来的には、収益分配の対象にもしたい（広告・採用手数料・バウンティなどからの還元）。
収益アイデア（将来）：
サイト全体の広告収入や、採用／案件成立時の紹介料の一部を
seed 運営
アイデア提供者
ソリューションを作ったエンジニア
に分配するイメージ。
3. 現時点の v1 方針（まず作るもの）
まずは 「アイデア掲示板サイト」 としてシンプルに始める。
収益化はリリース後に追加していく前提。
v1 でやる範囲：
アイデア投稿
アイデア一覧（掲示板）
「アイデアを実現してみた」ソリューション投稿
自分・他人のプロフィール表示
Googleログインによる認証（Firebase Auth）
将来用の情報（閲覧数・採用数・評価など）は、DB設計時に組み込んでいく想定（今は画面とフローを優先）。
4. 技術スタックの方向性
フロントエンド：TypeScript + Angular
認証：Firebase Authentication（まずは Google ログイン）
DB：PostgreSQL（ただし今はまだ手を付けず、API & フロントから設計）
バックエンド：
本線案は Java + Spring Boot + Firebase Auth連携 + PostgreSQL。
Firebase Auth 側で IDトークン発行 → Spring Boot で検証 → ユーザーIDとアプリ内ユーザー情報を紐づける構成を想定。
ただし現フェーズでは：
「一旦 DB・バックエンドは置いておき、フロントと画面構成から作る」 方針に変更済み。
5. 画面構成（seed v1）
ユーザーが挙げてくれた「欲しい画面」をベースに整理。
5-1. 掲示板（トップ）
URL：/
役割：
投稿されたアイデアの一覧表示。
各アイデアに対するソリューション数や投稿者情報も確認。
要素：
説明文、+ アイデアを投稿する ボタン（/ideas/new へ）
フィルタ／ソート（カテゴリ・難易度・ステータスなど）
アイデアカード一覧
タイトル
一行サマリ
カテゴリ
難易度
ステータス
投稿者（アイコン＋名前 → ポップ表示 → 他人のアカウントページへリンク）
ソリューション数
「詳細見る」や「このアイデアでソリューション投稿」ボタン
5-2. アイデア作成画面
URL：/ideas/new
役割：
アイデア提供者が seed に「種」を投稿する。
主な入力項目：
タイトル
一行サマリ
課題（Problem）
解決イメージ（Solution）
カテゴリ
公開レベル（オープン / 課題のみ公開）
難易度（★1〜5）
想定プラットフォーム（任意）
連絡先リンク（任意）
5-3. ソリューション投稿画面
URL：/solutions/new?ideaId=xxx
役割：
エンジニアが「実際に作ってみたもの」を投稿。
主な入力項目：
対象アイデア（タイトルなどを表示）
ソリューション名（任意）
何を作ったか（説明）
リンク（デモURL / GitHub 等、必須）
使用技術（任意）
5-4. マイページ
URL：/me
役割：
自分のプロフィールと、投稿したアイデア・ソリューションを一覧。
プロフィール内容：
アイコン
アカウント名
ID
プロフィール文
Twitter(X)リンク、GitHubリンク、ポートフォリオリンク など
タブ構成（案）：
「アイデア」タブ：自分が投稿したアイデア一覧
「ソリューション」タブ：自分が投稿したソリューション一覧
5-5. 他の人のアカウントページ
URL：/users/:id
役割：
他ユーザーのプロフィール＆投稿を見るページ。
内容：
マイページの閲覧版（編集なし）
プロフィール＋アイデア＋ソリューション一覧
追加要素：
掲示板などで投稿者のアイコン／名前をクリックすると
ポップで簡易プロフィール（アイコン・名前・一言・Twitter/GitHubリンク等）
「このユーザーのページへ」リンク
5-6. ログイン画面
URL：/login
役割：
Googleアカウントによるログイン。
要素：
seed の説明（ログインすると何ができるか）
Googleでログイン ボタン
Firebase Auth の Google Sign-in を呼ぶ
6. 共通ヘッダー仕様
全ページ共通で表示。
左側：
（将来）サービスアイコン
サービス名：seed
ページタイトル（掲示板 / アイデア作成 / ソリューション投稿 / マイページ など）
右側：
通知ボタン（ベルアイコン）
未ログイン時：
「ログイン」ボタン
ログイン時：
自分のアカウントアイコン
クリックでメニュー（マイページ／ログアウト）
7. 今後の進め方（フロントから）
現時点の方針：
Angular で画面（ページ）とルーティングを先に作る
中身はダミーデータで動かしながら UI を固める
その後、バックエンド/API/DB を繋ぎ込んでいく
さらにその後、閲覧数・評価・ポイントなどを足していき、
将来の収益モデル・就活用メトリクスに繋げる
